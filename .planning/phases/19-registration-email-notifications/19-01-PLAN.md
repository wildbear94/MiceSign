2. **V16__extend_notification_log_for_registration.sql** (Production, MariaDB):
   MariaDB에서 unnamed FK를 DROP하려면 information_schema에서 이름을 찾아야 함. 안전하게 하기 위해 FK constraint name을 직접 조회하는 대신, MariaDB의 `ALTER TABLE ... DROP FOREIGN KEY` 구문 사용. V1에서 FK 이름을 명시하지 않았으므로 자동 생성됨. MariaDB 규칙에 따라 테이블명_ibfk_N 형식.

   ```sql
   -- notification_log 테이블을 등록 알림 지원을 위해 확장
   
   -- Step 1: recipient_id FK 제거 후 nullable로 변경 (per D-26)
   -- MariaDB auto-generated FK name: notification_log_ibfk_1
   SET @fk_name = (SELECT CONSTRAINT_NAME FROM information_schema.TABLE_CONSTRAINTS 
                    WHERE TABLE_SCHEMA = DATABASE() 
                    AND TABLE_NAME = 'notification_log' 
                    AND CONSTRAINT_TYPE = 'FOREIGN KEY' 
                    AND CONSTRAINT_NAME LIKE '%ibfk%' LIMIT 1);
   
   ALTER TABLE notification_log DROP FOREIGN KEY notification_log_ibfk_1;
   ALTER TABLE notification_log MODIFY recipient_id BIGINT NULL;
   ALTER TABLE notification_log ADD CONSTRAINT fk_notification_log_recipient 
       FOREIGN KEY (recipient_id) REFERENCES `user`(id);
   
   -- Step 2: registration_request_id 컬럼 추가 (per D-27)
   ALTER TABLE notification_log ADD COLUMN registration_request_id BIGINT NULL 
       AFTER document_id;
   ALTER TABLE notification_log ADD INDEX idx_notification_registration (registration_request_id);
   ```

   참고: `SET @fk_name` 쿼리는 실제로는 사용하지 않음 (MariaDB에서 prepared statement 없이 변수로 DROP FK 불가). 대신 `notification_log_ibfk_1`을 직접 사용. 만약 FK 이름이 다르면 수동 확인 필요.

   실제 마이그레이션 파일은 아래 내용만 포함:
   ```sql
   -- notification_log: 등록 알림 지원 확장
   -- recipient_id를 nullable로 변경 (미등록 신청자는 user가 아님)
   ALTER TABLE notification_log DROP FOREIGN KEY notification_log_ibfk_1;
   ALTER TABLE notification_log MODIFY recipient_id BIGINT NULL;
   ALTER TABLE notification_log ADD CONSTRAINT fk_notification_log_recipient 
       FOREIGN KEY (recipient_id) REFERENCES `user`(id);
   
   -- registration_request_id 컬럼 추가
   ALTER TABLE notification_log ADD COLUMN registration_request_id BIGINT NULL 
       AFTER document_id;
   ALTER TABLE notification_log ADD INDEX idx_notification_registration (registration_request_id);
   ```

3. **V7__extend_notification_log_for_registration.sql** (Test, H2):
   H2는 MariaDB와 FK 구문이 다름. H2에서는 `ALTER TABLE ... DROP CONSTRAINT` 사용. H2 unnamed FK는 자동생성됨.
   
   H2에서는 unnamed FK를 직접 DROP하기 어려우므로, 다른 접근법 사용:
   - `ALTER TABLE notification_log ALTER COLUMN recipient_id SET NULL;` (H2는 이것만으로 nullable 변경 가능)
   - `ALTER TABLE notification_log ADD COLUMN registration_request_id BIGINT NULL;`
   
   ```sql
   -- H2: notification_log 등록 알림 지원 확장
   ALTER TABLE notification_log ALTER COLUMN recipient_id BIGINT NULL;
   ALTER TABLE notification_log ADD COLUMN registration_request_id BIGINT NULL;
   CREATE INDEX idx_notification_registration ON notification_log(registration_request_id);
   ```
