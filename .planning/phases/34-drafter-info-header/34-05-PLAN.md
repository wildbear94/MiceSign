<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → DocumentDetailResponse JSON | Trusted server-controlled fields (drafterName, departmentName, positionName, formData JSON); server-side access already gated by `getDocument` 4-check |
| browser → JSON.parse(doc.formData) | Server-stored data; could theoretically be malformed if a future bug allowed corrupt JSON, but DocumentService validates at write time |
| client → React render | XSS via name fields; React text-children escape protects |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-05-01 | Tampering | malformed `formData` JSON crashes detail page | mitigate | Inline `try { JSON.parse(...).drafterSnapshot ?? null } catch { return null }` in DocumentDetailPage isolates parse failure — falls back to legacy mode (live + badge). UI-SPEC Test 3 covers this rendering path. |
| T-34-05-02 | Information Disclosure | drafter dept/position visible in header | accept | Same fields already in `DocumentDetailResponse` — visible to anyone passing `getDocument` 4-check. No new exposure. UI-SPEC accent-forbidden inside header (purely presentational), no new interactive surface. |
| T-34-05-03 | Tampering | client-side XSS via spoofed dept/name | mitigate | All header values rendered as React text children — auto-escaped. Phase 34 introduces zero new HTML/DOM injection vectors. |
| T-34-05-04 | Repudiation | header shows wrong drafter (cache vs live) | accept | DRAFT mode reads from `existingDoc` (server response) or `authStore.user` (own profile — non-spoofable). SUBMITTED mode reads server-stamped snapshot. Header cannot show another user's data unless the BE response itself is wrong. |
| T-34-05-05 | DoS | new doc with no authStore user → blank header | mitigate | Default values `?? ''` and `?? null` prevent React rendering errors. UI-SPEC validates that the header gracefully shows empty strings (acceptable UX — user knows their own dept/position by context, and login flow always populates user before reaching the editor). |
</threat_model>
