<objective>
Fix the latent type-mismatch bug between the FE `DocumentDetailResponse` interface and the BE `DocumentDetailResponse` record. The FE currently declares `drafter: DrafterInfo` (nested) while the BE serializes flat fields (`drafterName`, `departmentName`, `positionName`). `DocumentDetailPage.tsx` L228 reads `doc.drafter.name`, which is `undefined.name` at runtime — production crash latent in current code (RESEARCH Pitfall 6, CONTEXT D-G1~G4).

This plan lands BEFORE any new feature work so subsequent plans (34-04 component, 34-05 integration) can rely on the corrected flat shape.

Purpose: Prevent the `DrafterInfoHeader` integration from inheriting a buggy data contract. The header's fallback path (D-D4 legacy fallback) reads `doc.drafterName` / `doc.departmentName` / `doc.positionName` — these MUST be flat for the contract to type-check.

Output: `types/document.ts` aligned to BE flat record + `DocumentDetailPage.tsx` L228 fix + meta-grid 기안자 cell deletion (UI-SPEC §"Position note for DocumentDetailPage" mandates removal to avoid duplication with the new header in subsequent plans). NOTE: the meta-grid 기안자 cell deletion is deferred to Plan 34-05 to keep this plan tightly scoped to the bug fix; this plan only **fixes** the L228 access (does not delete the cell).
</objective>
