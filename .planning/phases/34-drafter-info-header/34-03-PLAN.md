// Phase 34: drafter snapshot 박제 (D-C1, D-C2, D-A3)
User drafter = document.getDrafter();
String departmentName = drafter.getDepartment() != null
        ? drafter.getDepartment().getName() : null;
String positionName = drafter.getPosition() != null
        ? drafter.getPosition().getName() : null;
try {
    java.util.Map<String, Object> body;
    String currentFormData = content.getFormData();
    if (currentFormData == null || currentFormData.isBlank()) {
        body = new java.util.LinkedHashMap<>();
    } else {
        body = objectMapper.readValue(
                currentFormData,
                new com.fasterxml.jackson.core.type.TypeReference<java.util.LinkedHashMap<String, Object>>() {});
    }
    java.util.Map<String, Object> snapshot = new java.util.LinkedHashMap<>();
    snapshot.put("departmentName", departmentName);
    snapshot.put("positionName", positionName);  // null 허용 (D-C4) — key present, value JSON null
    snapshot.put("drafterName", drafter.getName());
    snapshot.put("draftedAt", document.getSubmittedAt().toString()); // ISO LocalDateTime
    body.put("drafterSnapshot", snapshot);
    content.setFormData(objectMapper.writeValueAsString(body));
    documentContentRepository.save(content);
} catch (JsonProcessingException e) {
    // D-C7 (Q2=A): 트랜잭션 전체 롤백 — submit 흐름 차단. RESEARCH 권고 옵션 B(swallow+WARN) 거절.
    log.error("Failed to serialize drafterSnapshot for document {}: {}",
            document.getId(), e.getMessage());
    throw new RuntimeException("DOC_SNAPSHOT_FAILED", e);
}

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/v1/documents/{id}/submit | Untrusted action by drafter; existing `loadAndVerifyOwnerDraft` validates ownership + status before snapshot |
| DocumentService → ObjectMapper | Internal — only writes server-assembled values (User entity fields), never client-supplied content into the snapshot |
| DocumentService → DB JSON column | Existing trust boundary — formData is already raw JSON from client; the snapshot block ADDS server-controlled keys, does not modify client keys |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-03-01 | Tampering | drafter snapshot post-creation | mitigate | D-C5 immutability is enforced by NOT writing drafterSnapshot anywhere except submit. Approve/reject/withdraw paths do not touch formData (verified by absence of `setFormData` in those service methods). Test C (snapshotImmutableAfterStatusChange) gates this. |
| T-34-03-02 | Information Disclosure | dept/position name leak in JSON | accept | Same data already flows via `DocumentDetailResponse.{departmentName, positionName}` flat fields — zero new exposure surface. ASVS V4 access control unchanged (existing `getDocument` 4-check gates form_data return). |
| T-34-03-03 | Tampering | client overwrites drafterSnapshot via direct API | mitigate | Client cannot send `drafterSnapshot` because (a) submit endpoint accepts no body for form_data updates, (b) update endpoint only allows status=DRAFT — once SUBMITTED, content is locked (DOC-04), (c) D-C5 immutability + (d) backend put() always overwrites if client-injected key existed in DRAFT formData. |
| T-34-03-04 | Denial of Service | Jackson exception aborts submit | accept (D-C7 explicit choice) | Q2=A user decision: data-consistency over availability. Submit fails fast and rolls back; user retries. Logged at error level for ops visibility. |
| T-34-03-05 | Tampering | malformed pre-existing formData breaks parse | mitigate | `objectMapper.readValue(currentFormData, ...)` throws `JsonProcessingException` → caught → D-C7 rollback path. Existing CreateDocumentRequest validation gate must already reject malformed JSON; an attacker storing malformed JSON via update would have to bypass that — out of this phase's scope. |
| T-34-03-06 | Repudiation | snapshot creation not audited | accept | Existing audit-log entry at L314~319 (`DOC_SUBMIT`) already records the SUBMITTED transition. Adding a separate audit row for snapshot creation would duplicate per NFR-03 ("COUNT=1 per action"). |
