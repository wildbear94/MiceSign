  <action>
    Fix 6 backend issues from verification and code review:

    **1. Remove gdriveFileId from AttachmentResponse (Verification gap + WR-05 / D-10)**

    In `AttachmentResponse.java`, remove the `gdriveFileId` field from the record. The record should be:
    ```java
    public record AttachmentResponse(
        Long id,
        Long documentId,
        String originalName,
        Long fileSize,
        String mimeType,
        LocalDateTime createdAt
    ) {}
    ```

    In `DocumentAttachmentMapper.java`, add `@Mapping(target = "gdriveFileId", ignore = true)` is NOT needed since the field no longer exists in the target record. However, the mapper auto-generates based on field name matching -- since gdriveFileId no longer exists in the target, MapStruct will simply skip it. No mapper change needed unless compilation fails. Verify by compiling.

    The `getAttachmentMetadata()` method in `DocumentAttachmentService.java` (line 266-269) returns `AttachmentResponse` which is used by `DocumentController.downloadAttachment()` to get `metadata.mimeType()`, `metadata.originalName()`, and `metadata.fileSize()`. None of these depend on gdriveFileId, so removing it from the DTO is safe. The download endpoint gets the actual gdriveFileId through `downloadAttachment()` which reads from the entity directly.

    **2. Fix audit log JSON injection (CR-01)**

    In `DocumentAttachmentService.java`, replace all manual JSON string concatenation with Jackson ObjectMapper. Inject `ObjectMapper` via constructor (Spring Boot auto-configures one). Replace these three occurrences:

    - Line 115-116 (uploadAttachment): `"{\"file\":\"" + file.getOriginalFilename() + "\"}"` -> use `objectMapper.writeValueAsString(Map.of("file", file.getOriginalFilename()))`
    - Line 172-173 (uploadFiles batch): same pattern
    - Line 196-197 (downloadAttachment): `"{\"file\":\"" + attachment.getOriginalName() + "\"}"` -> use `objectMapper.writeValueAsString(Map.of("file", attachment.getOriginalName()))`

    Add `private final ObjectMapper objectMapper;` field and constructor parameter. Spring Boot provides ObjectMapper as a bean.

    Wrap `writeValueAsString` in a private helper method to handle the checked `JsonProcessingException`:
    ```java
    private String toJson(Map<String, Object> data) {
        try {
            return objectMapper.writeValueAsString(data);
        } catch (JsonProcessingException e) {
            return data.toString(); // fallback
        }
    }
    ```

    **3. Fix deleteAttachment parameter order swap (WR-01)**

    In `DocumentController.java` line 189, the call is:
    `attachmentService.deleteAttachment(user.getUserId(), attachmentId);`

    But service signature is `deleteAttachment(Long attachmentId, Long userId)`.

    Fix by swapping arguments:
    `attachmentService.deleteAttachment(attachmentId, user.getUserId());`

    **4. Fix FileInputStream resource leak (WR-02)**

    In `GoogleDriveConfig.java` lines 55-58, wrap in try-with-resources:
    ```java
    try (FileInputStream fis = new FileInputStream(credFile)) {
        GoogleCredentials creds = GoogleCredentials
                .fromStream(fis)
                .createScoped(SCOPES);
        log.info("Using service account key file: {}", credentialsPath);
        return creds;
    }
    ```

    **5. Add null-safe filename validation (WR-03)**

    In `DocumentAttachmentService.java`, at the beginning of the `validateFile()` method (before the size check), add:
    ```java
    String originalName = file.getOriginalFilename();
    if (originalName == null || originalName.isBlank()) {
        throw new BusinessException("FILE_NAME_INVALID", "파일 이름이 없습니다.");
    }
    ```

    **6. MIME type validation note (CR-02)**

    CR-02 suggests using Apache Tika for server-side MIME detection. Adding a new dependency (Tika) is out of scope for gap closure. However, add a defensive comment at line 95 documenting the limitation, and ensure the extension blocklist (which IS checked server-side) remains the primary security gate. The MIME type from client is used for Content-Type on download -- this is acceptable since the extension blocklist prevents dangerous file types regardless of MIME spoofing.
