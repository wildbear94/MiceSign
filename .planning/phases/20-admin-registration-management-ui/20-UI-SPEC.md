#### Modal Content by Status

**All statuses -- top section (applicant info):**

| Field | Display |
|-------|---------|
| 이름 | Text, `font-medium` |
| 이메일 | Text |
| 신청일 | Formatted date `YYYY-MM-DD HH:mm` |
| 상태 | Status badge (same colors as table) |

Layout: Label-value pairs. Label: `text-sm font-semibold text-gray-500 dark:text-gray-400`. Value: `text-sm text-gray-900 dark:text-gray-50`.

**PENDING status -- action section (D-10):**

Approve area:
- 부서 dropdown (`<select>`): required, populated from department tree API
- 직급 dropdown (`<select>`): required, populated from positions API
- 사원번호 text input: required, max 20 chars, `placeholder="예: EMP001"`
- Approve button: `bg-blue-600 hover:bg-blue-700 text-white h-11 px-4 rounded-lg text-sm font-medium`
- No confirmation dialog for approve (D-15)

Reject area:
- "거부" button (initially visible): `bg-red-600 hover:bg-red-700 text-white h-11 px-4 rounded-lg text-sm font-medium`
- On click: reveals textarea for rejection reason (D-18)
- Textarea: `min-h-[80px]`, placeholder "거부 사유를 입력해주세요 (10자 이상)", minLength 10 characters
- "거부 확인" button below textarea
- On "거부 확인" click: opens ConfirmDialog with title "신청 거부", message "이 신청을 거부하시겠습니까?", confirmVariant "danger" (D-20)

Button layout: Approve and Reject buttons side by side at modal bottom. Approve left, Reject right. Both `h-11`.

**APPROVED status -- read-only section (D-11):**

| Field | Display |
|-------|---------|
| 배정 부서 | Department name |
| 배정 직급 | Position name |
| 사원번호 | Employee number |
| 처리일 | Formatted date |

Note: Backend DTO may need extension for these fields (RESEARCH.md Open Question 1). If not available, omit and show processedAt only.

**REJECTED status -- read-only section (D-11):**

| Field | Display |
|-------|---------|
| 거부 사유 | Rejection reason text |
| 처리일 | Formatted date |

**EXPIRED/CANCELLED -- no additional section.** Just applicant info + status.

## Interaction States

### Loading States

| Context | Behavior |
|---------|----------|
| List loading | Centered spinner in table body area (D-27) |
| Approve in progress | Approve button shows `Loader2` spinner icon + disabled state (`disabled:opacity-50`) |
| Reject in progress | ConfirmDialog confirm button shows `Loader2` spinner + disabled (existing ConfirmDialog `isLoading` prop) |
| Sidebar badge loading | No spinner; show badge only after data loads |

### Error States

| Context | Behavior |
|---------|----------|
| List fetch error | Inline error message below tabs: "데이터를 불러올 수 없습니다." with retry button |
| Approve/reject API error | Error toast (D-25) |
| Already processed (409/conflict) | Error toast "이미 처리된 신청입니다." + invalidate list query to refresh (D-26) |
| Validation error (form) | Inline below each field, red text `text-xs text-red-600` |

### Modal State Transitions

```
[Modal opens] -> mode: 'view'
  |
  +--> [Approve button clicked] -> validate form -> call approve API -> success toast -> close modal
  |
  +--> [거부 button clicked] -> mode: 'reject' (shows textarea)
         |
         +--> [거부 확인 clicked] -> validate minLength -> show ConfirmDialog
                |
                +--> [ConfirmDialog confirmed] -> call reject API -> success toast -> close modal
                +--> [ConfirmDialog cancelled] -> return to mode: 'reject'
         |
         +--> [취소/Cancel clicked] -> mode: 'view' (hides textarea)
```

## Accessibility

| Requirement | Implementation |
|-------------|----------------|
| Modal focus trap | Match ConfirmDialog pattern: focus trapped within modal, first focusable element receives focus on open |
| Modal close | ESC key closes modal (match existing pattern) |
| Table sort | `aria-sort` attribute on sortable column headers (match UserTable pattern) |
| Tab navigation | Tabs use `role="tablist"`, each tab `role="tab"`, `aria-selected` for active |
| Form labels | All form inputs have associated `<label>` elements |
| Loading states | Buttons use `disabled` attribute + `aria-busy="true"` during API calls |
| Color contrast | All badge text meets WCAG 2.1 AA (4.5:1 ratio) -- verified by existing UserTable badge pattern |
| ConfirmDialog | Inherits existing a11y: `role="dialog"`, `aria-modal="true"`, `aria-labelledby` |

## Discretion Decisions (Claude's Choices)

Decisions made for areas marked as "Claude's Discretion" in CONTEXT.md:

| Area | Decision | Rationale |
|------|----------|-----------|
| Reject reason min length | 10 characters | Prevents meaningless input like "X" while not being overly restrictive |
| Approve button style | `bg-blue-600 hover:bg-blue-700 text-white` (left position) | Matches project primary action pattern (blue = constructive) |
| Reject button style | `bg-red-600 hover:bg-red-700 text-white` (right position) | Matches project destructive action pattern (red = destructive) |
| Modal size | `max-w-[500px]` | Matches UserFormModal for visual consistency |
| Tab API optimization | TanStack Query `staleTime: 30_000` for sidebar badge query | Prevents re-fetch on every page navigation while keeping data reasonably fresh |
| Loading spinner | Centered in table body area, using `Loader2` from lucide-react with `animate-spin` | Consistent with existing button loading patterns |
| Status badge colors | Yellow/Green/Red/Gray matching UserTable STATUS_BADGE pattern | Direct reuse of established visual language |
| PENDING badge API | Fetched on sidebar mount, `staleTime: 30s`, uses list API with `status=PENDING&size=1` and reads `totalElements` | Minimal API cost, no polling |
