# Phase 36 — UI Design Contract

> Visual and interaction contract for the **양식 필드 한 줄 최대 3개 레이아웃** feature spanning **two distinct UI surfaces**:
> 1. **Builder** — admin assigns fields to row groups inside `SchemaFieldEditor` (per-field "행 위치" selector + visual grouping in `FieldCard` + live `FormPreview` reflection).

- Intra-row horizontal gap (renderer, md+): `gap-4` (16px) — new for this phase

**Row group indicator color cycle (left border on FieldCards):**

To help admin visually identify which fields belong to the same row group, FieldCards belonging to the same `rowGroup` value share a colored left border (`border-l-4`). Colors cycle through a 4-color palette to support up to 4 row groups visible simultaneously without visual collision:

| Cycle index | Tailwind class (light) | Tailwind class (dark) | Hex (light) |
|-------------|------------------------|-----------------------|-------------|
| 0 (rowGroup 1) | `border-l-blue-400` | `dark:border-l-blue-500` | `#60A5FA` |
| 1 (rowGroup 2) | `border-l-emerald-400` | `dark:border-l-emerald-500` | `#34D399` |
| 2 (rowGroup 3) | `border-l-violet-400` | `dark:border-l-violet-500` | `#A78BFA` |
| 3 (rowGroup 4) | `border-l-amber-400` | `dark:border-l-amber-500` | `#FBBF24` |
| 4+ (rowGroup 5+) | (cycle restarts at blue-400) | | |

**No left border** is applied for fields with `rowGroup === undefined` (the default — single-row vertical stack behavior).

**Mapping rule:** `cycleIndex = (rowGroup - 1) % 4`. RowGroup numbering is 1-indexed (matches admin mental model: "1행", "2행" rather than "0행").

**Why 400-shade left borders (not 500/600):** 400 shades are visible but recessive — they signal grouping without competing for attention with the FieldCard's own content (label, type badge, action buttons in 500/600 shades). Tested mentally against `bg-white` (light) and `bg-gray-900` (dark): `border-l-blue-400` vs `bg-white` ≈ 3:1 boundary contrast (sufficient for non-text decoration per WCAG 1.4.11 Non-text Contrast).

**Accent reserved for:**
1. Selected row-position selector button (`bg-blue-600 text-white`)
2. Row-group left-border indicator on FieldCards (4 cycling 400-shades)
3. Project-wide CTAs (existing — `focus:ring-blue-500`, `bg-blue-600` save/submit buttons)
4. **Forbidden** anywhere else in this phase (no decorative blue text, no blue backgrounds beyond #1~#3)

**Destructive reserved for:**

2. Existing project destructive actions (delete confirmations, withdrawal)

## Layout Contract

### A. Renderer Layout (DynamicCustomForm + DynamicCustomReadOnly)

**Top-level structure preserved** — `<form className="space-y-4">` (Edit) / `<div className="p-4 space-y-4">` (ReadOnly) unchanged. The change is in **how `schema.fields` is iterated**: instead of mapping each field to its own block, fields are first grouped into rows.

**Row grouping algorithm (executor reference):**

```ts
// Pseudocode — actual implementation in DynamicCustomForm / DynamicCustomReadOnly.
// Walks schema.fields in declaration order. Same-numbered consecutive rowGroup
// values that contain ONLY non-wide types form one row. Wide types
// (textarea/table) ALWAYS render as single-row regardless of rowGroup value.

type RowGroup =
  | { kind: 'single'; field: FieldDefinition }    // wide type OR rowGroup undefined
  | { kind: 'grid'; rowGroup: number; cols: 1 | 2 | 3; fields: FieldDefinition[] };

const WIDE_TYPES = new Set(['textarea', 'table']);

function groupFieldsByRow(fields: FieldDefinition[]): RowGroup[] {
  const result: RowGroup[] = [];
  let i = 0;
  while (i < fields.length) {
    const f = fields[i];
    if (WIDE_TYPES.has(f.type) || f.rowGroup === undefined) {
      result.push({ kind: 'single', field: f });
      i++;
      continue;
    }
    // collect consecutive same-rowGroup non-wide fields
    const groupNumber = f.rowGroup;
    const bucket: FieldDefinition[] = [];
    while (
      i < fields.length &&
      fields[i].rowGroup === groupNumber &&
      !WIDE_TYPES.has(fields[i].type)
    ) {
      bucket.push(fields[i]);
      i++;
    }
    // hard cap 3 (defensive — Zod validates upstream)
    const cols = Math.min(bucket.length, 3) as 1 | 2 | 3;
    result.push({ kind: 'grid', rowGroup: groupNumber, cols, fields: bucket.slice(0, 3) });
  }
  return result;
}
```

**Render template per group:**

```tsx
{groups.map((g) => {
  if (g.kind === 'single') {
    return (
      <div key={g.field.id}>
        <DynamicFieldRenderer field={g.field} {...} />
      </div>
    );
  }
  // g.kind === 'grid'
  const colsClass =
    g.cols === 1 ? 'md:grid-cols-1' :
    g.cols === 2 ? 'md:grid-cols-2' :
    'md:grid-cols-3';
  return (
    <div
      key={`row-${g.rowGroup}-${g.fields[0].id}`}
      role="group"
      aria-label={t('templates.rowLayout.rowGroupAriaLabel', { number: g.rowGroup })}
      className={`grid grid-cols-1 ${colsClass} gap-4`}
    >
      {g.fields.map((f) => (
        <div key={f.id} className="min-w-0">
          <DynamicFieldRenderer field={f} {...} />
        </div>
      ))}
    </div>
  );
})}
```

| Property | Specification |
|----------|---------------|
| Outer iteration container | `<form className="space-y-4">` (Edit) / `<div className="p-4 space-y-4">` (ReadOnly) — **unchanged** |
| Single-row group container | `<div>` wrapping `<DynamicFieldRenderer>` — **unchanged** from current behavior |
| Multi-field row group container | `<div role="group" aria-label="N행 그룹" className="grid grid-cols-1 md:grid-cols-{N} gap-4">` |
| Mobile (`<768px`) | All row groups render as 1-col vertical stack (`grid-cols-1`) — wide field forced-single still renders 1-col (no visual difference vs. mobile multi-field group) |
| Desktop (`≥768px`) | Row groups render `md:grid-cols-1` / `md:grid-cols-2` / `md:grid-cols-3` per group's `cols` value |
| Per-cell wrapper | `<div className="min-w-0">` — `min-w-0` is REQUIRED to prevent grid blowout when a child input has implicit min-width (e.g., long select option text). Without this, a `<input>` inside CSS grid can refuse to shrink below its content width |
| Cell vertical alignment | Default (`items-stretch`) — labels align to top of cell, inputs align below labels naturally; cells of unequal label-line-count remain visually aligned at top |
| Cell horizontal gap | `gap-4` (16px) — between fields in same row |
| Inter-row-group vertical gap | `space-y-4` (16px) — same gap as before, applied between all top-level groups (single AND grid) |
| Conditional-hidden field in row | The hidden field's wrapper renders `display:none` (matches existing `DynamicCustomForm.tsx` L247 pattern). CSS grid auto-collapses the empty cell — sibling cells reflow naturally to fill remaining columns. |
| Wide field forced-single | `textarea` / `table` always render in their own grid-less wrapper `<div>` regardless of `rowGroup` value (algorithm short-circuits) |
| Long-input overflow | Each cell uses `min-w-0` (see above) — ensures `<select>`, `<input type="text">` shrink to cell width. Text content inside read-only cells uses `whitespace-pre-wrap break-words` (existing pattern) |

**Why `grid grid-cols-1 md:grid-cols-N` (vs `flex md:flex-row`):**
- CSS grid handles unequal-height children gracefully (label + input + error message of varying heights). Flexbox row would cause cells to stretch to tallest; grid keeps them in tracks of equal height by default which is cleaner.
- Grid auto-collapse on `display:none` children is built-in; flex would leave dead space without `flex-grow:0` retrofit.
- Tailwind `md:grid-cols-{N}` is well-tested, no custom CSS, no `@apply`.

**Why NOT `inline-flex` per cell:**
- Inputs need full cell width (`w-full` already applied by `DynamicFieldRenderer`); inline-flex would require `flex-1` on cells which is more verbose than grid columns.

**Conditional rule × row group interaction (D-G4 regression scope):**

| Scenario | Behavior | Implementation note |
|----------|----------|---------------------|
| Field A in row group 1 has `condition` → `hide` | Cell A renders `display:none`; cells B, C in same row remain in their grid columns | Grid `display:none` cell collapses; remaining cells DO NOT reflow into vacated column (CSS grid behavior — empty track stays). This is acceptable: "consistent column positions" preserves admin's intended visual layout. |
| All fields in row group hidden | Entire row group wrapper still renders (empty grid). Acceptable; rare in practice; Phase 24.1 already has `display:none` patterns for hidden-but-form-state-preserved fields | If checker flags as visually empty, consider unmounting the entire `<div role="group">` when `g.fields.every(f => hiddenFields.has(f.id))` — DEFERRED implementation detail; not required by D-G4. |

**FormPreview reflection (D-A3):**

`FormPreview.tsx` already accepts `fields: SchemaField[]` (L11) and renders via `PreviewFieldRenderer`. To reflect row groups in real-time:
1. `FormPreview` MUST adopt the same `groupFieldsByRow()` algorithm as the renderer (both consume `SchemaField[]` / `FieldDefinition[]`).
2. Existing wrapper `<div className="space-y-4">` (L84) becomes the row-group iterator wrapper (preserves vertical gap between groups).
3. Multi-field groups wrap in `<div role="group" className="grid grid-cols-1 md:grid-cols-{N} gap-4">`.

### C. ReadOnly Layout (DynamicCustomReadOnly)

The same row grouping algorithm applies to ReadOnly, BUT:
- ReadOnly cells render `<div><dt>{label}</dt><dd>{value}</dd></div>` (existing `ReadOnlyField` pattern in `DynamicCustomReadOnly.tsx` L89~166).
- `table` and `textarea` cells in ReadOnly are wide and force-single-row — same rule.
- The ReadOnly grid uses `gap-4` matching the Edit grid for visual consistency.

**Backward compat (D-D1):** When `schemaSnapshot` was captured BEFORE Phase 36 ships, `field.rowGroup` is undefined for every field. The `groupFieldsByRow()` algorithm short-circuits the first branch (`f.rowGroup === undefined`) and emits one `{ kind: 'single' }` per field — visually identical to current `space-y-4` vertical stack. **Zero layout shift on legacy SUBMITTED docs.**

## Data Model Contract

**SchemaField extension (D-F1 option i — committed in this UI-SPEC):**

```ts
// File: frontend/src/features/admin/components/SchemaFieldEditor/types.ts
// File: frontend/src/features/document/types/dynamicForm.ts

export interface SchemaField {            // builder type
  id: string;
  type: SchemaFieldType;
  label: string;
  required: boolean;
  config: SchemaFieldConfig;
  rowGroup?: number;                      // NEW — 1-indexed, undefined = single-row vertical stack
}

export interface FieldDefinition {        // renderer type (mirrors backend record)
  id: string;
  type: FieldType;
  label: string;
  required: boolean;
  config?: FieldConfig;
  rowGroup?: number;                      // NEW — same shape, optional, JSON-serialized as `"rowGroup": N` or omitted
}
```

**Why option (i) and NOT (ii) `rows: Array<{fieldIds[]}>` or (iii) wrapper `'row'` element type:**

| Criterion | Option (i) `rowGroup?: number` | Option (ii) separate `rows[]` | Option (iii) wrapper element |
|-----------|--------------------------------|--------------------------------|-------------------------------|
| Backward compat (Phase 24.1 schemaSnapshot) | Trivial — undefined fields render legacy way | Requires migration — old snapshots have no `rows` field | Requires migration — element types differ |
| Field reordering (Phase 21 up/down chevrons) | Works as-is — moving a field changes its position; same-rowGroup neighbors regroup naturally | Requires bidirectional sync between `fields[]` and `rows[]` → bug surface | Requires reordering wrapper children → complex |
| Conditional rule fieldId references (Phase 24) | Unchanged — rules reference `field.id`, ignore `rowGroup` | Unchanged | **Breaks** — wrapper has no fieldId; nested fieldId paths needed |
| Calculation rule fieldId references (Phase 25) | Unchanged | Unchanged | Same break as above |
| Zod validation complexity | Add 1 optional field to `fieldDefinitionSchema` | New top-level schema + cross-validation between `rows.fieldIds` and `fields.id` | Recursive schema + element type discrimination |

**Zod schema delta (executor reference):**

```ts
// templateImportSchema.ts — fieldDefinitionSchema delta
const fieldDefinitionSchema = z
  .object({
    id: z.string().min(1),
    type: fieldTypeSchema,
    label: z.string().min(1),
    required: z.boolean(),
    config: fieldConfigSchema.optional(),
    rowGroup: z.number().int().positive().optional(),  // NEW — 1-indexed
  })
  .strict();

// schemaDefinitionSchema delta — add hard-cap-3 refinement + wide-type guard
const WIDE_TYPES_FOR_VALIDATION = new Set(['textarea', 'table']);

const schemaDefinitionSchema = z
  .object({
    version: z.number().int().positive(),
    fields: z.array(fieldDefinitionSchema),
    conditionalRules: z.array(conditionalRuleSchema).optional(),
    calculationRules: z.array(calculationRuleSchema).optional(),
  })
  .strict()
  .refine(
    (s) => {
      // Wide types must NOT have rowGroup
      return s.fields.every(
        (f) => !WIDE_TYPES_FOR_VALIDATION.has(f.type) || f.rowGroup === undefined,
      );
    },
    { message: 'templates.rowLayout.zodWideTypeError' },  // i18n key for client-side error display
  )
  .refine(
    (s) => {
      // Consecutive same-rowGroup non-wide fields must not exceed 3
      let i = 0;
      while (i < s.fields.length) {
        const rg = s.fields[i].rowGroup;
        if (rg === undefined || WIDE_TYPES_FOR_VALIDATION.has(s.fields[i].type)) {
          i++;
          continue;
        }
        let count = 0;
        while (
          i < s.fields.length &&
          s.fields[i].rowGroup === rg &&
          !WIDE_TYPES_FOR_VALIDATION.has(s.fields[i].type)
        ) {
          count++;
          i++;
        }
        if (count > 3) return false;
      }
      return true;
    },
    { message: 'templates.rowLayout.zodCapExceededError' },  // i18n key
  );
```

## Force-Single-Row Guard (Builder, D-C4)

When admin changes a field's type to `textarea` or `table` while `rowGroup` is set, OR drags a wide field into a row group:

**Behavior:**
1. `rowGroup` is silently set to `undefined` (consistency invariant).
2. A toast notification fires: `t('templates.rowLayout.wideTypeAutoSingleToast')` — body: `"이 필드는 한 줄을 차지하는 필드라 행 그룹에서 빠집니다."` (5-second auto-dismiss; existing `sonner` toast pattern from `SchemaFieldEditor.tsx` L67).

4. The expanded "행 위치" selector is hidden entirely (it's only shown for non-wide types per the conditional in §B above).
