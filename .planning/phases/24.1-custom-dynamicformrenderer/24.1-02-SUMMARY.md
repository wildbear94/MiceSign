# Phase 24.1 Plan 02: 사용자측 동적 폼 렌더러 핵심 컴포넌트 Summary

admin `PreviewFieldRenderer` 의 케이스별 렌더링 로직을 user `FieldDefinition` 타입 기반의 공유 `DynamicFieldRenderer` 로 추출했고, table 필드는 `useFieldArray` 기반의 `DynamicTableField` 로 분리했다. admin↔user 타입 차이를 흡수하는 `adaptSchemaField` adapter 도 함께 작성하여 Plan 04 의 admin FormPreview 회귀 통합을 위한 타입 브리지를 마련했다.

## What Was Built

### Task 1 — DynamicFieldRenderer + DynamicTableField stub (commit `5fd9ec8`)

`DynamicFieldRenderer.tsx` (213 lines) — 7개 비-table 필드 타입을 처리하는 단일 컴포넌트:

- `text`, `textarea`, `number` (`valueAsNumber: true`), `date`, `select`: react-hook-form `register` 를 통해 edit 모드 입력
- `staticText`: `config.content` 를 whitespace-pre-wrap 으로 표시 (입력 없음)
- `hidden`: 렌더 없음 (값은 form state 에서만 유지)
- `table`: `DynamicTableField` 에 위임 (edit 모드만, readOnly 는 Plan 03 범위)
- `mode='readOnly'` 분기: 라벨 + 텍스트 값. select 의 경우 option label 매핑, 빈 값은 `'-'`.
- edit 모드에서 `register` 미주입 시 명시적 throw 로 통합 누락 조기 발견.

`DynamicTableField.tsx` 빈 stub (`return null`) — Task 2 에서 본 구현으로 덮어쓸 수 있도록 최소 타입 시그니처만 충족.

## Deviations from Plan

### [Rule 3 - Blocking] PLAN 의 interface 가정을 실제 타입에 맞춰 정정

**발견:** Task 1 시작 직전 `frontend/src/features/document/types/dynamicForm.ts` 확인 결과, PLAN 의 `<interfaces>` 블록에 기재된 user `FieldDefinition` 형태가 실제 정의와 다음과 같이 달랐다:

| PLAN 가정 | 실제 정의 |
|-----------|-----------|
| `field.options?: OptionItem[]` (최상위) | `field.config?.options?: OptionItem[]` (config 안) |
| `field.columns?: ColumnDefinition[]` (최상위) | `field.config?.columns?: ColumnDefinition[]` (config 안) |
| `ColumnDefinition.options` (최상위) | `ColumnDefinition.config?.options` (config 안) |
| `field.required?: boolean` | `field.required: boolean` (non-optional) |
| staticText 본문: `config.text` | 실제: `config.content` |

**조치:** PLAN 의 코드 스니펫을 그대로 작성하면 다수의 TS2339/TS2353 에러가 발생했을 것이므로, 모든 옵션/컬럼 접근을 `field.config?.options` / `field.config?.columns` / `col.config?.options` 로 정정. staticText 본문은 `config.content` 로 정정. 동작 의미는 PLAN 과 동일.

**근거:** 실제 타입은 admin `SchemaFieldConfig` 와 거의 동일하게 `config` 안에 옵션/컬럼을 보관하도록 정의되어 있다. PLAN 의 가정은 RESEARCH 시점의 추정과 실제 dynamicForm.ts 사이의 드리프트로 보이며, 본 plan 의 D-04/D-05 추출 원칙(단일 렌더러)에는 영향이 없다.

### [Rule 3 - Blocking] admin TableColumnType 'checkbox' 누락 처리

**발견:** admin `TableColumnType` 에 `'checkbox'` 가 존재하지만 user `FieldType` 에는 없음. adapter 에서 강제 cast 시 런타임 동작 불일치.

**조치:** `mapColumnType` 함수로 `'checkbox' → 'text'` 폴백 명시 + 주석으로 24.1 범위 외임을 기록.

### [Rule 3 - Blocking] useFieldArray + 동적 name 의 타입 시그니처

**발견:** `useFieldArray<TFieldValues, TName>` 의 `TName` 은 `Path<TFieldValues>` (리터럴 dot-path) 를 요구하지만 `field.id` 는 런타임 string 이므로 직접 통과 불가.

**조치:** `control as unknown as Control<Record<string, never[]>>` + `name as never` 로 타입 캐스팅 (런타임 동작 동일). PLAN 도 `control as any` 사용을 시사했음. 캐스팅을 unknown 경유로 해 ESLint 친화적으로 정리.
