**Researched:** 2026-04-13
**Domain:** 프론트엔드 동적 폼 렌더링 (CUSTOM 템플릿 기안 작성/조회)
**Confidence:** HIGH (코드베이스 1차 근거 기반)

## Summary

Phase 24.1은 관리자 양식 빌더(Phase 22~24)로 만든 CUSTOM 템플릿을 사용자 측에서 기안 작성/조회할 수 있도록 `DynamicCustomForm`(편집) / `DynamicCustomReadOnly`(읽기) 컴포넌트를 만들고 `TEMPLATE_REGISTRY`에 통합한다.

**핵심 발견 — 작업 범위가 CONTEXT.md 가정보다 줄어듦:**

1. **백엔드는 이미 완료되어 있다.** `DocumentContent.schemaDefinitionSnapshot` 컬럼, `DocumentDetailResponse.schemaDefinitionSnapshot` 필드, `DocumentService.createDocument/updateDocument`의 자동 snapshot 저장 로직(`templateSchemaService.resolveSchemaWithOptions`)이 모두 구현 완료 — 프론트엔드는 요청 payload에 schema를 실어보낼 필요가 **없다**. create/update 시 백엔드가 `ApprovalTemplate.schemaDefinition`을 읽어 자동 snapshot 저장한다.
2. **Phase 24 빌드 블로커(D-27)는 이미 해결되었다.** 커밋 `56d0066 fix(24): remove duplicate import in constants.ts`에서 수정됨. D-27 작업 항목은 제거해도 된다.
3. **프론트엔드 TypeScript 타입만 확장하면 된다.** `DocumentDetailResponse`에 `schemaVersion?: number`, `schemaDefinitionSnapshot?: string` 필드 추가가 유일한 API 계약 변경 작업.
4. **CONTEXT.md D-02 재검토 필요.** 모든 CUSTOM 양식은 `templateCode === 'CUSTOM'`이 아니라 **각자 고유한 code**(예: `VAC_REQ_2026`)와 `isCustom=true` 플래그를 갖는다. 라우트 `/documents/new/:templateCode`와 `existingDoc.templateCode`만으로 식별 가능. 별도 `templateId`/`customTemplateId` 파라미터 불필요.

**Primary recommendation:** TEMPLATE_REGISTRY 조회 실패 + `isCustom=true`(또는 동등한 신호)일 때 `DynamicCustomForm`/`DynamicCustomReadOnly`로 fallback 분기. `templateApi.getTemplateSchema(code)`로 작성 모드 schema를 가져오고, 편집/조회 모드는 `existingDoc.schemaDefinitionSnapshot`을 파싱해 사용. react-hook-form + `schemaToZod` + `evaluateConditions`/`executeCalculations` 실시간 평가 조합으로 admin FormPreview와 동일한 UX 구현.

## Key Findings

### 1. 백엔드 schemaSnapshot 인프라 완성 [VERIFIED: DocumentService.java:131-182, DocumentContent.java:27, DocumentMapper.java:31-32]

백엔드는 이미 CUSTOM 문서 작성 시 자동으로 schema snapshot을 저장한다. `DocumentService.createDocument` / `updateDocument` 둘 다 다음 로직 포함:

```java
if (template.getSchemaDefinition() != null) {
    content.setSchemaVersion(template.getSchemaVersion());
    String resolvedSchema = templateSchemaService.resolveSchemaWithOptions(
            template.getSchemaDefinition());
    content.setSchemaDefinitionSnapshot(resolvedSchema);
}
```

**주목할 점:** `resolveSchemaWithOptions()`는 단순히 저장된 JSON을 그대로 넣는 것이 아니라 **옵션 테이블까지 해석해 확정된 스키마**를 만든다. 즉 프론트엔드가 snapshot을 별도로 보낼 필요도, 만들 필요도 없다.

→ **D-10 재해석:** 프론트엔드는 신규 작성 진입 시 `templateApi.getTemplateSchema(code)`로 렌더링용 schema를 가져오기만 하면 되고, 저장 payload(`CreateDocumentRequest`, `UpdateDocumentRequest`)는 기존대로 `title`/`bodyHtml`/`formData`/`approvalLines`만 보내면 된다. 백엔드가 `templateCode`로 template을 조회해 자동 snapshot.

`DocumentDetailResponse`는 이미 `schemaVersion`, `schemaDefinitionSnapshot` 필드를 포함한다([DocumentDetailResponse.java:21-22](backend/src/main/java/com/micesign/dto/document/DocumentDetailResponse.java)). 프론트엔드 TypeScript 타입만 확장하면 편집/조회가 가능하다.

### 2. `PreviewFieldRenderer`는 admin 타입에 결합 + table은 플레이스홀더 [VERIFIED: PreviewFieldRenderer.tsx:2, :84-142]

현재 `PreviewFieldRenderer`는 `import type { SchemaField } from '../SchemaFieldEditor/types'`로 admin 타입을 사용한다. 또한 table 케이스는 2행 고정 + 모든 입력 `disabled`인 플레이스홀더(L109 `{[0, 1].map((rowIdx) => (…))}`, L116 `<input ... disabled ...>`)다.

**추출 시 필요한 변경:**
1. `SchemaField` → `FieldDefinition`(user 타입)으로 prop 타입 교체
2. `config.placeholder` 등 접근을 `config?.placeholder ?? ''`로 optional-safe 변경
3. table 케이스를 사용자 입력 가능 + useFieldArray 통합 구조로 전면 재작성 (추가/삭제 버튼, 실제 value/onChange 바인딩)
4. admin 쪽에서도 같은 컴포넌트를 쓰도록 adapter 작성 — **option A:** admin이 `FieldDefinition`으로 직접 state 관리, **option B:** `adaptSchemaFieldToFieldDefinition()` 유틸로 래핑

**권장:** admin SchemaField와 user FieldDefinition의 구조적 차이가 크지 않으므로 **option B (adapter)** 로 빠르게 통합하되, table만큼은 editable 모드를 새로 만들고 admin의 disabled 모드는 prop 플래그(`editableTable={false}`)로 분기.

### 3. TEMPLATE_REGISTRY 패턴과 CUSTOM 분기 [VERIFIED: templateRegistry.ts, App.tsx:71, DocumentEditorPage.tsx:214-237, DocumentDetailPage.tsx:114-115]

현재 라우트는 `/documents/new/:templateCode` (App.tsx L71) 하나이며 DocumentEditorPage에서:
```ts
const templateEntry = TEMPLATE_REGISTRY[resolvedTemplateCode];
if (!templateEntry && !isEditMode) { return <알 수 없는 양식>; }
const EditComponent = templateEntry?.editComponent;
```

DocumentDetailPage는:
```ts
const templateEntry = TEMPLATE_REGISTRY[doc.templateCode];
const ReadOnlyComponent = templateEntry?.readOnlyComponent;
```

**CUSTOM 분기 패턴 (권장):** CONTEXT.md D-01은 "CUSTOM 키 추가"를 명시하지만 `TEMPLATE_REGISTRY['CUSTOM']` 하나로는 양식별로 다른 `code`(예: `VAC_REQ_2026`)를 핸들링할 수 없다. 대신 **lookup 실패 시 fallback 분기**를 쓴다:

```ts
// DocumentEditorPage
const templateEntry = TEMPLATE_REGISTRY[resolvedTemplateCode];
const isDynamic = !templateEntry; // or: existingDoc에 schemaDefinitionSnapshot 존재
if (isDynamic) {
  return <DynamicCustomForm templateCode={resolvedTemplateCode}
                            initialData={...}
                            schemaSnapshot={existingDoc?.schemaDefinitionSnapshot}
                            onSave={handleSave} />;
}
```

또는 `TEMPLATE_REGISTRY`에 특수 키 `__CUSTOM_FALLBACK__`을 두고 `getTemplateEntry(code)`가 registry miss + `schemaSnapshot` 존재 시 이 fallback을 반환하게 할 수도 있다. 기획자(D-01)의 "단일 진입점" 의도는 **"한 종류의 렌더러"** 이지 "한 종류의 레지스트리 키"를 의미하지 않는다고 해석.

### 4. CONTEXT.md D-02 재검토 — `templateCode === 'CUSTOM'` 가정 문제 [VERIFIED: ApprovalTemplate.java:42, V8 migration, TemplateService.java]

D-02는 "templateCode === 'CUSTOM'인 모든 사용자 정의 양식"이라 썼지만, 실제로는 **CUSTOM 양식마다 고유한 code**(관리자 생성 시 입력)를 갖는다. 구분 플래그는 `ApprovalTemplate.isCustom: boolean`. 프론트엔드는 현재 이 플래그가 응답에 노출되지 않으므로 두 가지 신호 중 하나를 사용:

- **(a)** `TEMPLATE_REGISTRY[code]` miss — 하드코딩 6종이 아닌 모든 것
- **(b)** `DocumentDetailResponse.schemaDefinitionSnapshot` 존재 — 편집/조회 모드
- **(c)** `TemplateResponse.isCustom` 필드를 신설하여 응답에 추가 — list 화면 등에서 활용

신규 작성 모드에서는 (a)만 신뢰할 수 있다 (templateApi fetch 전). 단 이는 관리자가 하드코딩 템플릿 코드(GENERAL 등)와 충돌하는 이름을 만들지 않는다는 전제가 필요하다 — **백엔드 is_custom=true일 때 code가 6종과 겹치지 않도록 검증하는지 Plan 단계에서 재확인 필요**.

**권장:** (a) + (b)를 조합하고, Plan 단계에서 `TemplateResponse.isCustom` 노출을 작은 백엔드 보강으로 추가(다만 Phase 24.1이 frontend-only로 묶여 있다면 (a)+(b)만으로도 충분).

### 5. react-hook-form + conditionalRules/calculationRules 통합 패턴 [VERIFIED: ExpenseForm.tsx:63-73, FormPreview.tsx:17-22]

ExpenseForm은 `watch('items')` + `useEffect` + `setValue` 조합으로 auto-sum을 구현한다. 같은 패턴을 CUSTOM 폼에 적용:

```tsx
const form = useForm({ resolver: zodResolver(zodSchema), defaultValues });
const watchedValues = form.watch(); // 모든 필드 구독

// conditionalRules 실시간 평가
const { hiddenFields, requiredFields } = useMemo(
  () => evaluateConditions(schema.conditionalRules ?? [], watchedValues),
  [schema.conditionalRules, watchedValues]
);

// calculationRules 실시간 실행 + setValue 반영
useEffect(() => {
  const results = executeCalculations(schema.calculationRules ?? [], watchedValues);
  for (const [fieldId, value] of Object.entries(results)) {
    // 현재 값과 다를 때만 setValue (루프 방지)
    if (form.getValues(fieldId) !== value) {
      form.setValue(fieldId, value, { shouldValidate: false, shouldDirty: false });
    }
  }
}, [watchedValues, schema.calculationRules]);
```

**주의 1 (루프 방지):** `setValue`가 watch를 재트리거하지 않도록 `shouldDirty: false` + 값 비교. ExpenseForm은 `totalAmount` 필드에 대해 동일 이슈를 `useEffect([watchedItems, setValue])`로 관리함.

**주의 2 (숨김 필드 검증 제외):** `schemaToZod`는 기본적으로 모든 필드를 검증한다. D-16을 지키려면:
- **Option A:** `zodResolver`를 감싸는 custom resolver로 `hiddenFields` 통과 전 필드를 스킵
- **Option B:** submit 직전 `form.unregister(hiddenFieldId)` 호출 (단 값 보존과 충돌)
- **Option C:** `schemaToZod` 호출 시 동적으로 hiddenFields 필드를 `.optional()`로 wrap한 새 스키마를 `useMemo`로 생성 → zodResolver 재바인딩

**권장:** Option C. `schema`가 stable하고 `hiddenFields`만 변하므로 `useMemo` 의존성에 넣기 쉽고, `unregister` 없이 값 보존(D-19) 요건과 자연스럽게 호환된다.

**주의 3 (숨김 필드 값 보존):** react-hook-form은 기본적으로 unmount 시 값 유지 (`shouldUnregister: false` 기본). D-19 자동 만족. 단 배열(table) 필드는 주의 — 배열 필드 자체가 숨겨지면 `useFieldArray` 언마운트가 문제될 수 있으므로 렌더 시 `display: none`으로 숨기고 unmount하지 않는 전략 권장.

### 6. table 필드 행 추가/삭제 — `useFieldArray` [CITED: ExpenseForm.tsx:45, :132-182]

### DynamicCustomForm 골격

```tsx
export default function DynamicCustomForm({ documentId, initialData, onSave }: TemplateEditProps) {
  const { templateCode } = useParams();
  const existingSnapshot = initialData?.schemaSnapshot; // TemplateEditProps 확장 필요
  const { data: fetchedSchema } = useQuery({
    queryKey: ['template-schema', templateCode],
    queryFn: () => templateApi.getTemplateSchema(templateCode!),
    enabled: !existingSnapshot && !!templateCode,
  });
  const schema: SchemaDefinition | null = useMemo(() => {
    if (existingSnapshot) return JSON.parse(existingSnapshot);
    return fetchedSchema ?? null;
  }, [existingSnapshot, fetchedSchema]);

  if (!schema) return <Skeleton />;

  return <DynamicFormInner schema={schema} initialData={initialData} onSave={onSave} />;
}

function DynamicFormInner({ schema, initialData, onSave }) {
  const parsedFormData = initialData?.formData ? JSON.parse(initialData.formData) : {};
  const defaultValues = { ...buildDefaultValues(schema), ...parsedFormData };

  const form = useForm({ defaultValues });
  const watched = form.watch();

  const { hiddenFields, requiredFields } = useMemo(
    () => evaluateConditions(schema.conditionalRules ?? [], watched),
    [schema.conditionalRules, watched]
  );

  // 동적 Zod 스키마 (hiddenFields 제외)
  const zodSchema = useMemo(() => {
    const visibleSchema: SchemaDefinition = {
      ...schema,
      fields: schema.fields.map(f =>
        hiddenFields.has(f.id) ? { ...f, required: false } : f
      ),
    };
    return schemaToZod(visibleSchema);
  }, [schema, hiddenFields]);
  // Note: resolver는 useForm에 initial로 주고, 이후 form.setError 수동 관리도 고려
  // 간단한 경로: 매 렌더 useForm에 새 resolver가 들어가지 않으므로 submit 시점에 parse 수동 호출

  // calculationRules 실시간 반영
  useEffect(() => {
    const results = executeCalculations(schema.calculationRules ?? [], watched);
    for (const [fid, v] of Object.entries(results)) {
      if (form.getValues(fid) !== v) {
        form.setValue(fid, v, { shouldValidate: false, shouldDirty: false });
      }
    }
  }, [watched, schema.calculationRules]);

  const onSubmit = async (values) => {
    await onSave({
      title: values.__title ?? initialData?.title ?? '',
      bodyHtml: undefined,
      formData: JSON.stringify(values), // hidden 값 포함 보존
    });
  };

  return (
    <form id="document-form" onSubmit={form.handleSubmit(onSubmit)}>
      {/* title 입력 */}
      {schema.fields
        .filter(f => !hiddenFields.has(f.id))
        .map(f => <DynamicFieldRenderer key={f.id} field={f} mode="edit"
                   control={form.control} register={form.register}
                   dynamicRequired={requiredFields.has(f.id)} />)}
    </form>
  );
}
```

**동적 resolver 주의:** react-hook-form의 `resolver`는 `useForm` 호출 시점에 고정된다. hiddenFields 변경에 따라 동적으로 resolver를 바꾸려면 `useForm({ resolver: dynamicResolver })` 대신 **커스텀 resolver**를 만들어 클로저 내에서 `hiddenFieldsRef`를 참조:

```ts
const hiddenFieldsRef = useRef(hiddenFields);
hiddenFieldsRef.current = hiddenFields;

const customResolver = useCallback((values) => {
  const visibleSchema = { ...schema, fields: schema.fields.map(f =>
    hiddenFieldsRef.current.has(f.id) ? { ...f, required: false } : f) };
  return zodResolver(schemaToZod(visibleSchema))(values, undefined, {} as any);
}, [schema]);

const form = useForm({ resolver: customResolver, defaultValues });
```

### Title 필드 처리

기존 하드코딩 폼은 title을 별도 `<input {...register('title')}>` 로 가지며 `formData`와 분리한다. CUSTOM 폼에서도 동일 패턴 유지 — schema에 title 필드가 없다고 가정하고, DynamicCustomForm 내부에 별도 title input을 추가하고 `onSave({title: ..., formData: JSON.stringify(values)})` 형태로 저장.
