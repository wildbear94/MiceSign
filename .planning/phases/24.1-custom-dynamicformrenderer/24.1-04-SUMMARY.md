# Phase 24.1 Plan 04: 페이지 통합 및 admin FormPreview 회귀 Summary

admin `PreviewFieldRenderer` 를 `DynamicFieldRenderer` 위임 thin wrapper 로 리팩터링하고, `DocumentEditorPage` / `DocumentDetailPage` 에 CUSTOM 템플릿 fallback 분기를 추가했다. FormPreview.tsx 는 단 1바이트도 수정하지 않았고(선택 A), onSave payload 는 D-10 준수로 schema 를 포함하지 않는다. 기존 6개 하드코딩 템플릿은 그대로 TEMPLATE_REGISTRY lookup 으로 분기되어 회귀 없음.

## What Was Built

### Task 1 — PreviewFieldRenderer → DynamicFieldRenderer 위임 (commit `1c7d3f7`)

**선택 A: 미변경** — PreviewFieldRenderer 의 기존 prop 시그니처 `{ field, value, onChange, dynamicRequired }` 를 유지하면서 내부만 재작성. FormPreview.tsx 는 전혀 손대지 않음 (`git diff` 0 bytes 확인).

변경 내용 (`frontend/src/features/admin/components/FormPreview/PreviewFieldRenderer.tsx`, 161→88 lines):

- 기존 7개 필드 타입별 JSX 전체 제거
- `adaptSchemaFieldToFieldDefinition(field)` 로 admin `SchemaField` → user `FieldDefinition` 변환
- 필드별 미니 `useForm<Record<string, unknown>>({ defaultValues: { [field.id]: value } })` 생성
- `<FormProvider {...form}>` 로 감싸서 자식 `DynamicTableField` 의 `useFormContext` 호출이 성립하도록 함 (table 필드 지원 필수조건)
- 외부 `value` 변경 → 내부 `setValue` 역방향 동기화 (FormPreview 의 "미리보기 초기화" 버튼 지원)
- 내부 `form.watch()` → 외부 `onChange` 단방향 동기화 (구독 cleanup 포함)
- `field.type === 'hidden'` 조기 반환 유지

**table 필드 UX 변경:** 원본은 `[0, 1]` 고정 disabled 행만 렌더했으나, DynamicTableField 위임으로 editable 로 변경됨. 의도된 UX 개선 — 관리자가 실제 table 필드 동작(행 추가/삭제)을 미리 확인 가능. Plan 05 수동 회귀 체크리스트에 기록 권장.

### Task 2 — templateRegistry + DocumentEditorPage + DocumentDetailPage fallback (commit `11b7982`)

#### 2.1) `templateRegistry.ts`

기존 `TEMPLATE_REGISTRY` 와 6개 하드코딩 엔트리는 **건드리지 않고** 세 개 export 추가:

```ts
export const DYNAMIC_CUSTOM_EDIT = DynamicCustomForm;
export const DYNAMIC_CUSTOM_READONLY = DynamicCustomReadOnly;

export function isCustomTemplate(
  templateCode: string | undefined,
  hasSnapshot = false,
): boolean {
  if (hasSnapshot) return true;
  if (!templateCode) return false;
  return !(templateCode in TEMPLATE_REGISTRY);
}
```

#### 2.2) `DocumentEditorPage.tsx`

- import 에 `DYNAMIC_CUSTOM_EDIT` + `isCustomTemplate` 추가
- `schemaSnapshot = existingDoc?.schemaDefinitionSnapshot ?? null` 계산
- `useCustomRenderer` 판정: 편집 모드(`isEditMode`)는 스냅샷 존재로, 신규 작성은 `isCustomTemplate(resolvedTemplateCode)` 로 결정
- "알 수 없는 양식" 에러 UI 는 `!templateEntry && !useCustomRenderer && !isEditMode` 일 때만 표시 (CUSTOM fallback 진입 전 단락)
- `EditComponent = templateEntry?.editComponent ?? (useCustomRenderer ? DYNAMIC_CUSTOM_EDIT : undefined)`
- `initialData` 에 `schemaSnapshot` 전달 — 하드코딩 템플릿은 무시, `DynamicCustomForm` 만 사용
- **`handleSave` 는 완전 미변경** — payload `{ title, bodyHtml, formData }` 그대로 (D-10 준수)

#### 2.3) `DocumentDetailPage.tsx`

- import 에 `DYNAMIC_CUSTOM_READONLY` 추가
- `schemaSnapshot = doc.schemaDefinitionSnapshot ?? null` 계산
- `ReadOnlyComponent = templateEntry?.readOnlyComponent ?? (schemaSnapshot ? DYNAMIC_CUSTOM_READONLY : undefined)` — 스냅샷이 있으면 CUSTOM fallback, 없으면 기존 "알 수 없는 양식" UI 유지
- `<ReadOnlyComponent ... schemaSnapshot={schemaSnapshot} />` 로 prop 전달 (하드코딩 ReadOnly 는 prop 무시)

**검증:**
- `cd frontend && npx tsc --noEmit` → exit 0
- `grep -n "schemaSnapshot|schemaDefinition" DocumentEditorPage.tsx` → `existingDoc?.schemaDefinitionSnapshot`, `!!schemaSnapshot`, `initialData.schemaSnapshot` 3건 — 모두 initialData 경로, handleSave 경로 **없음** (D-10 준수)
- 기존 6개 템플릿 코드(GENERAL/EXPENSE/LEAVE/PURCHASE/BUSINESS_TRIP/OVERTIME)는 `templateEntry` truthy → fallback 진입 안 함
