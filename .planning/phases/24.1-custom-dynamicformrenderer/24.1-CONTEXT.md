# Phase 24.1: 사용자 측 동적 폼 렌더러 (DynamicFormRenderer) - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

관리자가 양식 빌더(Phase 22~24)로 만든 **CUSTOM 템플릿**(SchemaDefinition JSON)을 사용자가 실제로 기안 작성/조회할 수 있도록 동적 폼 렌더러를 구축한다.

**범위 내:**
- CUSTOM 템플릿 코드를 가진 문서의 작성/편집(`DocumentEditorPage`) 및 읽기 전용 표시(`DocumentDetailPage`)
- SchemaDefinition 기반 필드 렌더링(text, textarea, number, date, select, table, staticText, hidden)
- conditionalRules 실시간 평가(표시/숨김/필수)
- calculationRules 실시간 자동 계산
- schemaToZod 기반 유효성 검증
- table 필드 행 추가/삭제 입력 UX
- formData JSON 저장/로드 라운드트립
- 양식 변경 시에도 과거 문서가 정상 표시되도록 schema snapshot 저장

**범위 외 (Deferred):**
- 양식 변경 시 기존 DRAFT 문서 마이그레이션
- 사용자 측 양식 검색/즐겨찾기
- 신규 필드 타입 추가

</domain>

<decisions>
## Implementation Decisions

### 컴포넌트 통합 방식
- **D-01:** `TEMPLATE_REGISTRY`에 `CUSTOM` 항목 추가 — 단일 진입점으로 `DynamicCustomForm`(편집)과 `DynamicCustomReadOnly`(읽기 전용) 등록. 페이지 분기 최소화, 기존 패턴과 일관성 유지
- **D-02:** `templateCode === 'CUSTOM'`인 모든 사용자 정의 양식이 동일한 렌더러를 통해 처리됨. 양식 자체의 식별은 별도 식별자(`templateId` 또는 `customTemplateId`)로 구분
- **D-03:** `DocumentEditorPage`/`DocumentDetailPage`는 기존 레지스트리 lookup 흐름을 그대로 사용. CUSTOM 케이스에서만 schema 데이터를 추가 prop으로 전달

### 코드 재사용 전략
- **D-04:** 관리자 미리보기 컴포넌트를 공유 가능한 형태로 추출 — `frontend/src/features/document/components/dynamic/` 디렉터리 신설
- **D-05:** `PreviewFieldRenderer.tsx` → `DynamicFieldRenderer.tsx`로 추출/이동, admin `FormPreview`와 사용자측 `DynamicCustomForm` 모두에서 import. 한 곳에서 유지보수
- **D-06:** admin `FormPreview.tsx`는 추출된 컴포넌트를 import하도록 리팩터링(기능 변경 없음, 회귀 검증 필요)
- **D-07:** 기존 evaluateConditions/executeCalculations/schemaToZod 유틸은 위치 변경 없이 그대로 재사용

### 데이터 흐름 / 영속성
- **D-08:** **Schema snapshot 저장 정책** — document 생성 시 해당 시점의 `schemaDefinition` 전체를 document에 함께 저장. 이후 양식이 수정/삭제되더라도 과거 문서는 원본 schema로 정상 표시
- **D-09:** snapshot 저장 위치 — `document_content` 또는 `document` 테이블의 `schemaSnapshot` JSON 컬럼(researcher가 백엔드 스키마 확인 후 결정). 프론트엔드 관점에서는 document 응답에 `schemaDefinition` 필드가 포함되는 것을 가정
- **D-10:** **백엔드 자동 snapshot 저장 유지** — 백엔드 `DocumentService`가 `templateSchemaService.resolveSchemaWithOptions`를 통해 templateId로부터 자동으로 snapshot을 생성/저장한다. 프론트엔드는 `CreateDocumentRequest` / `UpdateDocumentRequest` 페이로드에 schema를 포함하지 **않는다**(확장 불필요). 편집/조회 시에는 document 응답의 `schemaDefinitionSnapshot` 필드를 파싱해서 사용한다. 신규 작성 진입 시에는 templateApi로 schema를 fetch하여 렌더링에만 사용하고, 저장 payload에는 포함하지 않는다.
- **D-11:** 편집/조회 시에는 document 응답에 포함된 schema snapshot 사용 (별도 fetch 없음)
- **D-12:** 사용자 입력값 저장 형식 — `formData`는 `{ [fieldId]: value }` 형태의 JSON 문자열. table 필드는 `{ [fieldId]: Array<{ [columnId]: value }> }`

### 유효성 검증
- **D-13:** `schemaToZod` 유틸 활용 + react-hook-form 통합 — 기존 `GeneralForm` 등 하드코딩 폼과 동일한 패턴
- **D-14:** 검증 실패 시 필드별 인라인 에러 메시지 표시(기존 패턴과 일관)
- **D-15:** 제출(submit) 시점에 `form.checkValidity()` 트리거 — `DocumentEditorPage`의 기존 흐름 그대로 사용
- **D-16:** conditionalRules에 의해 숨겨진 필드는 검증 대상에서 제외(hidden 상태에서 required 검증 발생 방지)

### conditionalRules / calculationRules 동작
- **D-17:** **실시간 평가** — 사용자 입력 시마다 `evaluateConditions` 호출하여 표시/숨김/필수 즉시 적용 (관리자 인터랙티브 미리보기와 동일 UX)
- **D-18:** calculationRules도 입력 시마다 `executeCalculations` 호출하여 결과 필드 자동 갱신. 결과 필드는 readOnly 표시
- **D-19:** 숨김 전환된 필드의 값은 formData에 그대로 보존(다시 표시될 때 복원). 단, 검증에서만 제외
- **D-20:** 평가는 클라이언트에서만 수행 — 백엔드는 schemaDefinition + formData를 그대로 저장(기존 정책 유지)

### 읽기 전용 모드
- **D-21:** **별도 컴포넌트** `DynamicCustomReadOnly` — 입력 컴포넌트와 다른 표시 패턴(라벨 + 값 텍스트, table은 표 형식)으로 결재 후 가독성 우선
- **D-22:** 읽기 전용에서도 conditionalRules에 의해 숨겨진 필드는 표시하지 않음(작성 시점의 실제 폼 상태 반영)
- **D-23:** 읽기 전용은 기존 하드코딩 ReadOnly 컴포넌트(GeneralReadOnly 등)와 동일한 레이아웃/타이포그래피 사용

### table 필드 입력 UX
- **D-24:** 컬럼 정의대로 셀 입력, 행 단위 추가/삭제 버튼 제공
- **D-25:** 각 컬럼 타입별 적절한 입력(text/number/date/select)
- **D-26:** 행 검증은 컬럼별 `required`를 따름. **빈 행은 저장 시점(onSubmit)에 자동 제거** — 모든 컬럼 값이 비어 있는 행을 `filter`로 걸러내고 그 결과를 JSON.stringify한다. 명시적 안내 UI는 현 단계에선 불필요.

### Phase 24 빌드 블로커 동시 수정
- **D-27:** `frontend/src/features/admin/components/SchemaFieldEditor/constants.ts`의 중복 `import type { SchemaFieldType }`(라인 3, 63)을 단일 import로 병합. `npm run build` 실패 해결. 본 Phase 작업 시작 직전 사전 작업으로 처리

### Claude's Discretion
- 디렉터리 구조의 정확한 위치(`features/document/components/dynamic/` vs 다른 후보)
- `DynamicFieldRenderer` 추출 시 admin↔user 호환을 위한 prop 인터페이스 설계
- 추출 후 admin `FormPreview` 회귀 검증 방법
- table 행 추가/삭제 버튼의 정확한 UI/위치
- readOnly 모드에서 table을 표/카드 중 어떤 형태로 표시할지
- 필드별 에러 메시지의 정확한 카피
- staticText / hidden 필드의 표시 방식

</decisions>

<specifics>
## Specific Ideas

- 사용자측 동적 폼은 관리자 인터랙티브 미리보기와 **동일한 UX 느낌**이어야 함 — 미리보기에서 본 그대로 작성 가능
- 양식이 나중에 수정되더라도 과거 결재 문서는 **그 시점의 모습 그대로 표시**되어야 함(전자결재 시스템의 핵심 요구)
- 기존 하드코딩 템플릿(GENERAL/EXPENSE/LEAVE/PURCHASE/BUSINESS_TRIP/OVERTIME)은 변경하지 않음 — CUSTOM은 추가 분기

</specifics>

## Existing Code Insights

### Reusable Assets
- **`evaluateConditions`** — 10개 비교 연산자 + show/hide/require/unrequire 평가 완성. 그대로 재사용
- **`executeCalculations`** — 계산 공식 실행 유틸. 입력 변경 시 호출
- **`schemaToZod`** — SchemaDefinition → Zod 변환. react-hook-form `zodResolver`에 직접 연결 가능
- **`PreviewFieldRenderer`** — 인터랙티브 필드 렌더러. 추출 후 admin/user 공유
- **`AutoSaveIndicator` / `useAutoSave`** — 기존 자동 저장 훅 그대로 재사용
- **`react-hook-form` + `zod`** — 기존 하드코딩 폼이 사용 중. 동일 의존성 그대로 사용
- **`ApprovalLineEditor`, `FileAttachmentArea`** — `DocumentEditorPage`에 이미 통합. 변경 불필요

### Established Patterns
- **TEMPLATE_REGISTRY**: 모든 폼은 `editComponent` + `readOnlyComponent` 페어 등록
- **react-hook-form + zod**: 모든 하드코딩 폼이 사용 중인 검증 패턴
- **`onSave({title, bodyHtml, formData})`**: 모든 EditComponent의 통일된 저장 콜백 시그니처
- **JSON 저장**: `formData`는 stringify하여 단일 JSON 컬럼에 저장
- **에러 토스트**: `sonner` 사용

### Integration Points
- `templateRegistry.ts`에 `CUSTOM` 키 추가 → `DocumentEditorPage`/`DocumentDetailPage`가 기존 lookup 흐름 그대로 사용
- `templateApi.getTemplate(code/id)` — schema fetch
- `documentApi.create/update` — `schemaSnapshot` 필드 전달 검증 필요(researcher가 백엔드 응답 구조 확인)

### Architectural Constraints
- 결재 문서는 제출 후 immutable — schema snapshot이 필수(소급 변경 방지)
- 백엔드는 schemaDefinition을 의미적으로 해석하지 않음 — 모든 평가는 클라이언트에서 수행
- 문서 본문(bodyHtml)과 폼 데이터(formData)는 분리 관리 — CUSTOM도 동일 정책

</code_context>

<deferred>
## Deferred Ideas

- **양식 변경 시 기존 DRAFT 문서 마이그레이션** — 사용자가 작성 중인 DRAFT 문서가 있을 때 양식이 수정되면 어떻게 처리할지(스냅샷 정책으로 기본 회피, 정밀 정책은 후속 phase)
- **사용자 측 양식 검색/즐겨찾기** — 양식이 많아질 때 필요. 현재는 TemplateSelectionModal로 충분
- **신규 필드 타입** — file, signature, user-picker 등은 Phase 24.1 범위 외
- **readOnly 모드에서 conditionalRules 평가 정확성** — 작성 시점의 실제 폼 상태가 아닌 다른 시나리오 표시는 별도 논의 필요
- **schemaSnapshot 마이그레이션 도구** — 기존 CUSTOM 문서에 schemaSnapshot이 없을 경우 백필링(현재 CUSTOM 문서가 없으므로 불필요)

</deferred>
