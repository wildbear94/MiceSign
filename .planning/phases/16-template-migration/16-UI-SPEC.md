## Layout Contract

### DocumentDetailPage Read-Only Branching (Modified)

Current (broken for dynamic templates):
```
ReadOnlyComponent exists?
  YES -> Render hardcoded ReadOnlyComponent
  NO  -> Show "알 수 없는 양식입니다." (gap)
```

After Phase 16:
```
ReadOnlyComponent exists AND document has NO schemaDefinitionSnapshot?
  YES -> Render hardcoded ReadOnlyComponent (legacy documents)
document has schemaDefinitionSnapshot?
  YES -> Render DynamicReadOnly with snapshot + formData
NEITHER?
  -> Show "알 수 없는 양식입니다."
```

Visual layout of DynamicReadOnly in detail page is identical to existing hardcoded read-only views:
- Container: `bg-white dark:bg-gray-900 border border-gray-200 dark:border-gray-700 rounded-xl p-6 mb-6` (existing wrapper)
- DynamicReadOnly renders inside this container with its own field layout

### DocumentEditorPage Edit Branching (Modified)

Current:
```
TEMPLATE_REGISTRY[templateCode] exists?
  YES -> Hardcoded EditComponent
  NO  -> DynamicForm
```

After Phase 16 (new documents from migrated templates):
```
Template has schema_definition?
  YES -> DynamicForm (regardless of TEMPLATE_REGISTRY presence)
  NO  -> TEMPLATE_REGISTRY[templateCode] -> Hardcoded EditComponent

Existing DRAFT documents (created before migration):
  TEMPLATE_REGISTRY[templateCode] -> Hardcoded EditComponent
  (preserved by checking document's schemaDefinitionSnapshot)
```

### Schema Loading State (in DynamicForm)

DynamicForm already handles schema loading internally (implemented in Phase 13). No additional loading UI needed.

```
+------------------------------------------+
| [spinner] 양식을 불러오는 중...              |  <- existing loading state
+------------------------------------------+
```

### Schema Load Error State (in DynamicForm)

```
+------------------------------------------+

| 없습니다. 페이지를 새로고침해 주세요.         |  <- existing error state
+------------------------------------------+
```

Users should not notice the migration. Documents created before and after migration must look and behave identically from the user's perspective. There are no migration-specific UI elements, banners, or notifications.

## Interaction Contract

### Read-Only View (DocumentDetailPage)

| Interaction | Behavior |
|-------------|----------|
| View legacy document (pre-migration, GENERAL/EXPENSE/LEAVE) | Hardcoded ReadOnlyComponent renders as before -- zero change |
| View new dynamic document (post-migration) | DynamicReadOnly renders from schemaDefinitionSnapshot + formData |
| View document with unknown template (no registry, no snapshot) | "알 수 없는 양식입니다." text displayed |

### Edit Mode (DocumentEditorPage)

| Interaction | Behavior |
|-------------|----------|
| Edit existing DRAFT (created before migration) | Hardcoded EditComponent renders as before -- zero change to formData structure |
| Create new document from migrated template | DynamicForm loads schema from API, renders dynamic form |
| Edit DRAFT created after migration | DynamicForm loads schema, pre-fills from saved formData |

### Loading States

| Component | Loading Behavior |
|-----------|-----------------|
| DynamicForm schema fetch | Existing skeleton pulse animation (3 lines) until schema loaded |
| DynamicReadOnly | Renders immediately from snapshot prop (no async loading) |
| DocumentDetailPage | Existing loading state while document data fetches |

### Error States

| Component | Error Behavior |
|-----------|---------------|
| DynamicForm schema fetch failure | "양식 스키마를 불러올 수 없습니다. 페이지를 새로고침해 주세요." in red text |
| DocumentDetailPage data fetch failure | Existing error handling (unchanged) |

## Visual Parity Checklist

This is the key UI validation contract for Phase 16. Each migrated template's dynamic rendering must be visually compared against its hardcoded counterpart.

| Template | Edit Mode Parity | Read-Only Parity | Notes |
|----------|-----------------|-------------------|-------|
| GENERAL | Dynamic textarea vs Tiptap rich text -- **known gap** (A1 from RESEARCH.md) | Field layout must match | Rich text loss accepted per RESEARCH.md Pitfall 1 |
| EXPENSE | Table with item rows, auto-sum total | Table layout, amount formatting | Intra-row calculation (qty * price) support must be verified (A2) |
| LEAVE | Select + date fields, conditional fields | Field values display | Date difference calculation limitation accepted (A3) |
| PURCHASE | New schema (no hardcoded reference) | New schema | Fields per PRD/FSD spec |
| BUSINESS_TRIP | New schema (no hardcoded reference) | New schema | Fields per PRD/FSD spec |
| OVERTIME | New schema (no hardcoded reference) | New schema | Fields per PRD/FSD spec |

**Verification method (D-05):** Manual side-by-side comparison in browser for GENERAL/EXPENSE/LEAVE. PURCHASE/BUSINESS_TRIP/OVERTIME verified against PRD/FSD field specifications.
